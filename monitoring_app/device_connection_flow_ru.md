# Подключение Устройств И Отправка Телеметрии В MVP

## 1. Общая Идея

В MVP устройство не регистрируется само.

Сначала его создает администратор клиента в системе, затем для устройства выпускается отдельный токен, после чего устройство начинает отправлять телеметрию в backend.

Главная схема такая:

```text
admin -> создает устройство -> выпускает токен -> записывает токен в устройство
устройство -> отправляет POST /ingest -> backend проверяет токен -> сохраняет телеметрию
```

Важно:

- пользовательский JWT и токен устройства — это разные вещи
- устройство работает только по своему `device token`
- backend сам определяет, какому устройству и какому tenant принадлежат входящие данные

## 2. Как Устройство Появляется В Системе

Устройство сначала должно быть заведено в платформе как сущность.

Поток такой:

1. Администратор логинится в систему по JWT.
2. Администратор создает устройство.
3. Backend создает запись в таблице `devices`.
4. Устройство становится известным платформе.

Пример того, что хранится в `devices`:

- `id`
- `tenant_id`
- `name`
- `type`
- `status`
- `created_at`

Это нужно, чтобы backend всегда понимал:

- какое именно устройство отправило данные
- какому клиенту это устройство принадлежит

## 3. Как Выдается Токен Устройству

Для MVP используется простой и безопасный вариант:

1. Администратор нажимает "выпустить токен"
2. Backend генерирует случайный секрет
3. В БД сохраняется только хэш токена
4. Сам токен показывается админу только один раз
5. Администратор вручную записывает токен в устройство

Таблица `device_tokens`:

- `id`
- `device_id`
- `token_hash`
- `created_at`
- `revoked_at`

Почему в базе хранится только хэш:

- при утечке БД злоумышленник не получает готовые рабочие токены
- это тот же подход, что и для паролей пользователей

## 4. Каким Должен Быть Токен

Для MVP токен должен быть:

- случайным
- длинным
- уникальным
- отдельным для каждого устройства

Например:

```text
iot_dv_3mY9jK8XnP2QvR7sT1wA6bC4dE5fH8kL
```

Рекомендация:

- генерировать минимум 32 байта случайных данных
- отдавать токен в Base64Url или hex-формате
- не использовать один токен на несколько устройств

## 5. Как Токен Попадает В Устройство

В MVP provisioning ручной.

То есть:

- backend показывает токен админу
- админ копирует токен
- админ записывает его в конфиг устройства или прошивку

Варианты хранения на устройстве:

- `config.json`
- переменная конфигурации
- flash / EEPROM / NVS
- secure storage, если устройство его поддерживает

Пример локальной конфигурации устройства:

```json
{
  "apiBaseUrl": "https://api.example.com",
  "deviceToken": "iot_dv_3mY9jK8XnP2QvR7sT1wA6bC4dE5fH8kL",
  "sendIntervalSeconds": 30
}
```

## 6. Как Устройство Отправляет Данные В Backend

Для MVP используется HTTP POST.

Типовой запрос:

```http
POST /ingest HTTP/1.1
Host: api.example.com
Authorization: Bearer iot_dv_3mY9jK8XnP2QvR7sT1wA6bC4dE5fH8kL
Content-Type: application/json
```

Body:

```json
{
  "message_id": "8b5c5dd4-8c61-4baf-b2b8-4bf9a45f1e8f",
  "timestamp": "2026-04-26T15:00:00Z",
  "temperature": 24.7,
  "humidity": 50.1,
  "payload": {
    "battery": 91,
    "signal": -68
  }
}
```

Устройство должно отправлять:

- Bearer token в заголовке `Authorization`
- JSON в body
- уникальный `message_id`
- `timestamp`
- измерения
- произвольный `payload`, если нужен

## 7. Что Делает Backend При Приеме Телеметрии

Когда приходит `POST /ingest`, backend должен:

1. Прочитать `Authorization: Bearer <token>`
2. Проверить, что токен существует
3. Проверить, что токен не отозван
4. Найти устройство по токену
5. Определить `device_id`
6. Определить `tenant_id`
7. Провалидировать JSON payload
8. Принять `message_id`
9. Сохранить сообщение

На этапе 1:

- сообщение пишется напрямую в `telemetry`

На этапе 2:

- сообщение публикуется в Kafka
- затем storage worker пишет его в `telemetry`

## 8. Как Backend Проверяет Токен

Так как в БД хранится не сам токен, а только его хэш, backend работает так:

1. Получает plaintext token из заголовка
2. Хэширует его тем же алгоритмом
3. Ищет запись в `device_tokens`
4. Проверяет, что `revoked_at` пустой
5. Если запись найдена, токен считается валидным

После этого backend знает:

- какое это устройство
- какому tenant принадлежит устройство

## 9. Должно Ли Устройство Отправлять `device_id`

Для MVP лучше не доверять `device_id`, если он приходит в body.

Правильный подход:

- устройство отправляет token и payload
- backend сам определяет `device_id` по token

Почему это важно:

- устройство не сможет притвориться другим устройством
- меньше шанс логических ошибок на клиенте
- backend остается единственным источником истины

Если `device_id` и передается, то только как вспомогательное поле для диагностики, но не как источник истины.

## 10. Как Часто Устройство Должно Отправлять Данные

Это зависит от типа устройства.

Типовые режимы:

- периодически, например каждые 10, 30 или 60 секунд
- по событию, например при изменении значения
- смешанный режим: heartbeat плюс событийные сообщения

Для MVP обычно достаточно:

- периодической отправки
- heartbeat раз в N секунд

Пример:

- температурный датчик на складе шлет данные каждые 30 секунд
- даже если значение почти не меняется, устройство все равно отправляет heartbeat

## 11. Как Устройство Должно Вести Себя При Ошибках

Устройство должно уметь обрабатывать типовые ответы backend:

- `200/201/202` — сообщение принято
- `400` — payload неправильный, повторять не нужно
- `401/403` — токен неверный или отозван
- `429` — слишком частые запросы, нужно подождать
- `503` — backend временно недоступен, нужен retry

Очень важное правило:

- при retry устройство должно отправлять тот же `message_id`

Иначе система не сможет надежно определить, что это повтор одного и того же сообщения.

## 12. Как Работает Дедупликация

Для дедупликации используется `message_id`.

Логика такая:

- каждое сообщение имеет UUID
- если устройство повторно отправляет тот же пакет
- backend или worker понимает, что это уже известное сообщение
- дубль не записывается повторно

На этапе 1:

- дедупликация может контролироваться при записи в БД

На этапе 2:

- дедупликацию обычно делает storage worker

## 13. Что Должно Быть В Конфиге Устройства

Минимальный набор настроек устройства:

- `apiBaseUrl`
- путь `/ingest`
- `deviceToken`
- интервал отправки
- таймаут запроса
- логика retry

Пример:

```json
{
  "apiBaseUrl": "https://api.example.com",
  "ingestPath": "/ingest",
  "deviceToken": "iot_dv_...",
  "sendIntervalSeconds": 30,
  "requestTimeoutSeconds": 10,
  "maxRetryCount": 5
}
```

## 14. Полный Жизненный Цикл Подключения Устройства

Полный путь от физического устройства до данных в UI:

1. Админ регистрируется в платформе
2. Создается `tenant`
3. Админ создает устройство в UI
4. Backend создает запись в `devices`
5. Админ выпускает токен устройства
6. Backend сохраняет хэш токена в `device_tokens`
7. Backend показывает plaintext token один раз
8. Админ записывает токен в конфиг устройства
9. Устройство запускается
10. Устройство делает `POST /ingest` с Bearer token
11. Backend проверяет token
12. Backend определяет `device_id` и `tenant_id`
13. Backend валидирует payload
14. Backend сохраняет телеметрию или отправляет ее в Kafka
15. Данные попадают в `telemetry`
16. Пользователь открывает UI
17. UI запрашивает устройства и историю телеметрии через `/api/*`
18. Backend фильтрует данные по `tenant_id`
19. Пользователь видит только свои устройства и свои данные

## 15. Какие Admin API Нужны Для Устройств

Для MVP разумно предусмотреть такие endpoint'ы:

- `POST /api/devices`
  - создать устройство
- `GET /api/devices`
  - список устройств текущего tenant
- `GET /api/devices/{id}`
  - получить устройство
- `POST /api/devices/{id}/tokens`
  - выпустить новый токен
- `POST /api/devices/{id}/tokens/revoke`
  - отозвать текущий токен
- `GET /api/devices/{id}/telemetry`
  - получить историю телеметрии

Логическое разделение должно быть таким:

- пользователь работает через JWT
- устройство работает через device token

Эти два контура аутентификации не должны смешиваться.

## 16. Как Делается Ротация Токена

Иногда токен нужно заменить:

- токен утек
- устройство перепрошили
- устройство скомпрометировано
- устройство передали другому клиенту

Правильный путь:

1. Админ выпускает новый токен
2. Новый токен записывается в устройство
3. Старый токен отзывается через `revoked_at`

Так безопаснее и проще управлять жизненным циклом секрета.

## 17. Что Делать Если Устройство Украли Или Токен Утек

Нужно:

1. отозвать токен
2. при необходимости пометить устройство как подозрительное
3. выпустить новый токен
4. переподключить устройство уже с новым токеном

После отзыва любой `POST /ingest` со старым токеном должен отклоняться.

## 18. Что Важно По Безопасности Именно Для MVP

Минимальный рекомендуемый набор:

- только HTTPS
- отдельный токен на каждое устройство
- хранить только хэш токена
- использовать `revoked_at` для отзыва
- включить rate limiting на `/ingest`
- не доверять `device_id` и `tenant_id` из body
- определять устройство только по token
- требовать `message_id` для идемпотентности

## 19. Что Можно Добавить Позже

Когда система вырастет, можно добавить:

- MQTT вместо или вместе с HTTP
- mTLS для устройств
- bootstrap endpoint с одноразовым кодом
- сертификаты устройств
- secure provisioning
- подпись payload
- локальный буфер на устройстве при offline
- batch upload нескольких сообщений

Но для MVP это обычно избыточно.

## 20. Рекомендуемый MVP Протокол Для Устройства

Устройство хранит:

- `baseUrl`
- `deviceToken`
- `sendIntervalSeconds`

Устройство делает:

- `POST /ingest`
- `Authorization: Bearer <deviceToken>`
- `Content-Type: application/json`

Пример body:

```json
{
  "message_id": "uuid",
  "timestamp": "ISO-8601 UTC",
  "temperature": 23.4,
  "humidity": 48.2,
  "payload": {
    "battery": 87
  }
}
```

Backend делает:

- проверку токена
- определение `device_id`
- определение `tenant_id`
- валидацию payload
- запись в `telemetry` или публикацию в Kafka

## 21. Самая Важная Мысль

Для MVP устройство не должно быть "умным клиентом".

Оно не должно:

- знать структуру БД
- знать `tenant_id`
- знать Kafka
- использовать user JWT
- принимать сложные архитектурные решения

Устройство должно уметь только:

- хранить свой токен
- собирать измерения
- отправлять HTTP POST на `/ingest`
- повторять запросы при временных ошибках
- использовать тот же `message_id` при retry

Это и есть правильная минимальная модель подключения устройств для MVP.
