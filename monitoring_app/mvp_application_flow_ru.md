# MVP Приложение Для Мониторинга IoT Устройств

## 1. Что Делает MVP

MVP представляет собой мультитенантную платформу мониторинга IoT-устройств.

Для каждого клиента или организации в системе существуют свои:

- `tenant`
- пользователи
- устройства
- токены устройств
- телеметрические данные

Главная идея такая:

1. Администратор регистрируется в системе.
2. Система создает `tenant` и первого пользователя с ролью `admin`.
3. Администратор создает устройства и выпускает для них токены.
4. Устройства отправляют телеметрию в backend.
5. Backend проверяет токен устройства, определяет `tenant_id` и `device_id`, затем сохраняет телеметрию.
6. Пользователи входят по JWT и видят только свои устройства и свои данные.

На первом этапе MVP прием данных пишет напрямую в PostgreSQL/TimescaleDB.

На следующем этапе между приемом данных и записью в базу появляется Kafka и workers.

## 2. Полная Картина Целиком

Ранний MVP:

```text
устройство -> /ingest -> API -> PostgreSQL/TimescaleDB -> /api/* -> UI
```

После добавления Kafka:

```text
устройство -> /ingest -> API -> Kafka -> worker -> PostgreSQL/TimescaleDB -> /api/* -> UI
```

Позже:

```text
новая телеметрия -> SignalR -> UI обновляется без refresh
worker оповещений -> alerts table -> SignalR -> UI показывает alert
```

## 3. Как Получаем Данные Из Устройств

Для MVP устройства отправляют данные по HTTP.

MQTT есть в плане, но не в первой версии.

Формат сообщений для MVP — JSON.

Каждое устройство аутентифицируется собственным Bearer-токеном.

Это не пользовательский токен, а отдельный токен устройства, который выпускается платформой и хранится в таблице `device_tokens` в виде хэша.

Каждое сообщение телеметрии должно содержать:

- `message_id` типа UUID
- `timestamp`
- телеметрические поля, например `temperature` и `humidity`
- произвольный `payload`

Пример:

```json
{
  "message_id": "8b5c5dd4-8c61-4baf-b2b8-4bf9a45f1e8f",
  "timestamp": "2026-04-26T15:00:00Z",
  "temperature": 24.7,
  "humidity": 50.1,
  "payload": {
    "battery": 91,
    "signal": -68
  }
}
```

Почему важен `message_id`:

- он обеспечивает идемпотентность
- если устройство повторно отправит тот же пакет из-за retry, дубль можно обнаружить и не записывать второй раз

## 4. Куда Сохраняются Данные

Основное хранилище — PostgreSQL с расширением TimescaleDB.

Основные таблицы:

- `tenants`
- `users`
- `devices`
- `device_tokens`
- `telemetry`
- позже: `alert_rules`
- позже: `alerts`

Таблица `telemetry` сделана как TimescaleDB hypertable по полю `timestamp`.

В `telemetry` хранятся:

- `device_id`
- `tenant_id`
- `timestamp`
- `message_id`
- `temperature`
- `humidity`
- `payload`

Ключевой момент:

`tenant_id` хранится прямо в телеметрии, чтобы выборки по арендатору были быстрыми и безопасными.

## 5. Как Работает MVP На Этапе 1

На этапе 1 Kafka еще нет.

Самый простой рабочий поток такой:

1. Устройство отправляет `POST /ingest`.
2. Backend проверяет токен устройства.
3. Backend определяет `device_id` и `tenant_id`.
4. Backend валидирует JSON payload.
5. Backend напрямую сохраняет запись в `telemetry`.

Эта первая версия важна тем, что быстро дает полностью рабочий вертикальный срез:

- устройства могут отправлять данные
- данные сохраняются в базе
- пользователи могут читать эти данные через UI

## 6. Как Система Развивается На Этапе 2

После этапа 1 ingest перестает писать прямо в базу и начинает публиковать события в Kafka.

Новый поток выглядит так:

1. Устройство отправляет `POST /ingest`.
2. API проверяет токен устройства.
3. API подставляет `tenant_id` и `device_id`.
4. API публикует событие в Kafka topic `telemetry.raw`.
5. Kafka распределяет сообщения по ключу `device_id`.
6. Storage worker читает `telemetry.raw`.
7. Worker проверяет `message_id` на дубликаты.
8. Worker записывает данные в PostgreSQL/TimescaleDB.

Зачем нужна Kafka:

- развязывает прием данных и запись в БД
- лучше переживает всплески нагрузки
- упрощает масштабирование
- позволяет добавлять новые consumers

## 7. Почему В Kafka Ключ Это `device_id`

Ключ партиционирования в Kafka должен быть `device_id`.

Это дает:

- сохранение порядка сообщений одного устройства
- предсказуемую обработку
- более простую дальнейшую агрегацию и обработку правил

## 8. Как Пользователи Получают Данные

Пользователи не работают с базой напрямую.

Они используют `/api/*`.

Поток такой:

1. Пользователь регистрируется или логинится.
2. Backend выдает JWT.
3. Frontend отправляет JWT в каждом запросе.
4. Backend извлекает `tenant_id` из JWT.
5. Каждый запрос фильтруется по `tenant_id`.

Это защищает от утечки данных между арендаторами.

Пользователь из tenant A никогда не должен увидеть устройства или телеметрию tenant B.

Типовые действия пользователя:

- получить список устройств
- открыть страницу устройства
- посмотреть историю телеметрии
- позже: увидеть alerts и управлять правилами

## 9. Как Выглядит Frontend MVP

Frontend планируется как React SPA.

Основные страницы MVP:

- login
- список устройств
- страница устройства
- график истории телеметрии

Позже добавятся:

- лента оповещений
- CRUD для правил оповещений

Типовой путь пользователя:

1. Пользователь входит в систему.
2. Пользователь видит все устройства своего tenant.
3. Пользователь открывает конкретное устройство.
4. Пользователь видит последние и исторические значения телеметрии.

## 10. Как Позже Появится Realtime

Realtime появляется на этапе 3 через SignalR.

До SignalR:

- UI получает данные обычными HTTP-запросами

После SignalR:

1. В систему приходит новая телеметрия.
2. Backend отправляет обновление через SignalR hub.
3. React-клиент получает событие.
4. Графики и карточки устройств обновляются без перезагрузки страницы.

## 11. Как Позже Будут Работать Оповещения

Оповещения относятся к этапу 4.

Появляются новые сущности:

- `alert_rules`
- `alerts`

Примеры правил:

- `temperature > threshold`
- устройство офлайн
- аномалия в `payload`

Поток такой:

1. Телеметрия попадает в систему.
2. Alerts worker читает поток.
3. Worker сравнивает сообщение с правилами.
4. Если правило сработало, создается запись в `alerts`.
5. Backend отправляет alert в UI.

То есть alerts — это автоматическая реакция на поток телеметрии.

## 12. Как Будет Определяться Offline

Это пока открытый архитектурный вопрос.

Возможные варианты:

- периодический sweep по `last_seen`
- Redis TTL плюс heartbeat
- таймер внутри Kafka consumer

Для MVP самый простой путь такой:

- смотреть на время последней телеметрии устройства
- если прошло слишком много времени, устройство считается offline

## 13. Модель Безопасности

### Безопасность Пользователей

- JWT-аутентификация
- хранение паролей в виде хэшей
- все запросы фильтруются по `tenant_id`

### Безопасность Устройств

- отдельный Bearer-токен на каждое устройство
- в БД хранится только хэш токена
- токен можно отозвать через `revoked_at`

### Безопасность Транспорта

- HTTPS обязателен
- rate limiting на `/ingest`
- если позже Kafka недоступна, `/ingest` должен возвращать `503`

## 14. Модель Надежности

План уже закладывает основные механизмы надежности:

- `message_id` для дедупликации
- `tenant_id` во всех важных местах
- Kafka key = `device_id`
- TimescaleDB hypertable для временных рядов
- worker-based асинхронную обработку
- health checks, метрики и tracing

## 15. Полный Жизненный Цикл Системы

Идеальный жизненный цикл платформы такой:

1. Администратор регистрируется.
2. Система создает `tenant`.
3. Администратор создает устройство.
4. Система выпускает device token.
5. Токен настраивается в устройстве.
6. Устройство отправляет телеметрию.
7. Backend принимает и сохраняет данные.
8. Пользователь входит по JWT.
9. UI показывает только данные текущего tenant.
10. Позже SignalR дает realtime-обновления.
11. Позже alerts worker создает alerts и отправляет их в UI.

## 16. Первая Реально Полезная Бизнес-Версия

Первая реально полезная бизнес-версия MVP включает:

- регистрацию пользователя
- JWT login
- создание устройства
- выпуск device token
- прием телеметрии через `/ingest`
- сохранение телеметрии в `telemetry`
- список устройств в UI
- страницу устройства и историю данных в UI

Это уже полноценная работающая система мониторинга.

Следующие полезные улучшения:

- Kafka между ingest и storage
- realtime UI через SignalR
- правила оповещений и alert feed

## 19. Краткий Итог

Планируемый MVP — это многослойная система мониторинга:

1. Сначала создаются tenant и пользователи.
2. Затем регистрируются устройства и выпускаются токены.
3. Устройства отправляют телеметрию по HTTP в формате JSON.
4. Backend аутентифицирует устройства и привязывает данные к нужному tenant.
5. Данные сохраняются в PostgreSQL/TimescaleDB.
6. Пользователи аутентифицируются по JWT и видят только свои данные.
7. Затем добавляются Kafka и workers для масштабируемой обработки.
8. SignalR добавляет realtime-обновления в UI.
9. Правила и alerts добавляют автоматический мониторинг и реакции системы.

То есть MVP начинается просто, но архитектура изначально готовится к росту.
